# React Native 다국어 앱에서 버전 파편화가 조용히 생기는 이유

_written by Claude-Code_

> 영어 버전인데 화면에 한국어가 나왔다.
> 그리고 `Sundaty`라는 요일이 있었다.

---

## 배경

React Native 앱이 한국어·영어 두 언어로 운영하면서 각 언어 버전을 별도 파일로 관리되는 구조였다. 기능을 추가할 때는 주로 한국어 버전에서 먼저 개발하고 영어 버전에 반영했다. 그러다 영어 버전에 개발이 중단된 채 남겨진 파일이 있었다.

- 영어 화면 안에 한국어 문구가 하드코딩된 채로 남아 있었다
- `Sundaty`, `Wedneswday` 같은 오타가 UI에 그대로 표시되고 있었다
- 한국어에서는 `Home2_1 → Home2_2` 순서로 진행하는데, 영어는 `Home2_1`을 건너뛰고 `Home2_2`로 바로 이동했다
- 이미 제거했다고 생각한 더미 저장 함수가 한국어 버전에만 살아 있었다

---

## 왜 파편화가 생기는가

### 병렬 파일 구조의 함정

한/영 버전을 별도 파일로 분리하면 처음에는 단순해 보인다. 그런데 기능이 추가되거나 수정될 때마다 두 파일을 동시에 수정해야 한다. 현실에서는 이 동기화가 자주 빠진다.

```
src/screens/ko/Home2_1Screen.tsx   ← 먼저 개발
src/screens/en/Home2_1Screen.tsx   ← 나중에 반영 (또는 잊혀짐)
```

한국어 파일을 완성한 직후에는 영어 파일도 업데이트하려는 의도가 있다. 하지만 다음 기능으로 넘어가면 직전 변경이 영어에 반영됐는지 확인하기 어렵다.

### 텍스트를 코드처럼 다루는 문제

`i18n` 라이브러리를 쓰지 않고 문자열을 직접 JSX 안에 넣으면, 텍스트는 언어 파일이 아닌 컴포넌트 파일에 박혀 있다. 나중에 언어를 바꿀 때 파일 전체를 훑어야 한다.

```tsx
// 영어 파일인데 한국어가 하드코딩되어 있는 상황
<Text>오늘의 측정 결과</Text>
<Button title="다음" />
```

이 경우 빌드 오류가 발생하지 않는다. 앱은 정상 실행되지만 화면은 잘못된 언어를 보여준다.

### Route는 누락되어도 앱이 죽지 않는다

`React Navigation`에서 특정 화면으로 `navigate`를 호출할 때, 해당 route가 스택에 없으면 보통 에러가 발생한다. 그런데 조건 분기나 화면 초기화 흐름 안에 숨어 있으면 테스트 중에 그 경로를 지나치지 않아 발견이 늦다.

```tsx
// 한국어: Home2_1 → Home2_2 (정상 흐름)
// 영어: Home2_2로 바로 이동 (Home2_1 누락)
navigation.navigate('Home2_2', { ... });
```

기능상 앱이 동작하기 때문에 문제로 인식되지 않는다. 하지만 두 버전이 다른 사용자 경험을 제공하고 있다.

---

## 파편화의 유형

### 1. 하드코딩된 텍스트

가장 흔하고 눈에 띄지 않는 유형이다. 영어 파일 안에 한국어 문자열이 그대로 남아 있는 경우로, 빌드나 lint 단계에서 잡히지 않는다.

### 2. 오타

복붙 과정에서 생기거나, 처음부터 잘못 타이핑된 문자열이 영어 전용 경로에 있어서 발견이 늦다. `Sundaty`, `Wedneswday`는 한국어 버전에는 해당 화면이 없거나 다른 방식으로 표시되기 때문에 오랫동안 발견되지 않았다.

### 3. Route 누락 또는 불일치

한국어에 있는 화면이 영어 스택에 없거나, 이동 순서가 다르다. 사용자 흐름이 언어마다 달라지는 결과를 낳는다.

### 4. 로직 불일치

한국어에는 적용된 검증, 성공 조건 판단, 저장 방식이 영어에는 반영되지 않은 경우다. 같은 기능인데 동작이 다르다.

### 5. 더미 코드 잔존

개발 중 테스트용으로 넣은 함수, 임시 값, 콘솔 로그가 한 버전에만 남아 있는 경우다. 동작에 영향을 주거나 불필요한 부수효과를 만든다.

## 탐지 방법

### 파일 쌍 직접 비교

한/영 파일을 나란히 열고 구조, 텍스트, route 이동, 사용 함수를 대조한다. IDE의 diff 기능이나 `git diff`를 활용한다.

```bash
# 두 파일의 구조적 차이를 빠르게 확인
diff src/screens/ko/Home2_1Screen.tsx src/screens/en/Home2_1Screen.tsx
```

### 한국어 문자 검색

영어 전용 디렉토리에서 한국어 유니코드 범위를 검색한다.

```bash
# 영어 디렉토리 안에서 한글 문자 탐지
grep -rn '[가-힣]' src/screens/en/
```

### Route 목록 비교

한/영 Navigation Stack에 등록된 화면 목록을 추출해서 비교한다. 한쪽에만 있는 route가 의도적 차이인지 누락인지 확인한다.

### 화면별 점검 체크리스트

```
[ ] 화면에 표시되는 모든 텍스트가 해당 언어로 되어 있는가?
[ ] 버튼, 라벨, 안내 문구에 오타가 없는가?
[ ] 이동하는 route가 해당 언어 스택에 존재하는가?
[ ] 한국어 버전과 동일한 로직(검증, 저장, 이동)이 적용되어 있는가?
[ ] 더미 함수, 임시 주석, 미사용 import가 없는가?
```

## 수정 기준: 한국어를 원본으로 삼는다

두 버전이 다를 때 어느 쪽이 맞는지 판단 기준이 필요하다. 이 프로젝트에서는 한국어 버전을 먼저 개발하는 흐름이므로, **한국어 버전을 기준(source of truth)** 으로 정했다.

수정 순서는 다음과 같다.

1. 한국어 버전의 해당 화면이나 로직을 확인한다
2. 영어 버전과 다른 부분을 모두 식별한다
3. 영어 버전을 한국어 구조에 맞게 업데이트한다
4. 텍스트만 영어로 교체한다

이렇게 하면 로직 동기화와 번역 작업이 명확히 분리된다.

## 예방책

### 공통 로직 추상화

한/영 버전이 같은 동작을 해야 하는 함수는 언어에 무관한 공통 모듈로 분리한다. 측정값 저장, 동기화 트리거, 유효성 검증이 대표적인 대상이다.

```tsx
// 언어 버전에 무관하게 공통 사용
import { saveMeasurementForSync } from '@/shared/sync';
import { triggerMeasurementSyncInBackground } from '@/shared/sync';
```

공통 함수가 있으면 한쪽에만 로직이 바뀌는 상황 자체가 줄어든다. 이 동기화 방식의 설계 원칙은 [[health-data-sync-strategy|의료 데이터 동기화 설계 전략]]에서 다룬다.

### 언어 중립 컴포넌트 설계

텍스트를 props로 받는 컴포넌트를 만들면, 한/영 화면이 같은 컴포넌트를 공유하면서 텍스트만 다르게 전달할 수 있다.

```tsx
<MeasurementGuide
  title={t('guide.title')}
  instruction={t('guide.instruction')}
/>
```

i18n 라이브러리를 도입하지 않더라도, props 기반으로 텍스트를 외부에서 주입하는 패턴만으로도 하드코딩을 크게 줄일 수 있다.

### 기능 추가 시 두 버전을 함께 처리

새 기능을 한국어 버전에 추가할 때 영어 버전도 같은 커밋 또는 같은 PR에서 처리하는 것을 원칙으로 삼는다. 나중에 몰아서 하면 누락이 생긴다.

---

## 마무리

기능을 빠르게 검증하려면 한 언어 버전에서 먼저 개발하는 것이 자연스럽다. 문제는 그 이후다. 영어 버전을 업데이트하는 작업은 새 기능처럼 눈에 띄지 않기 때문에 뒤로 밀리고, 쌓이고, 잊힌다.

주기적으로 두 버전을 나란히 놓고 비교하는 습관, 그리고 공통 로직을 언어에 무관한 층으로 분리하는 설계가 파편화를 막는 현실적인 방법이다.

React Native 앱의 실행 구조와 번들러 동작 방식은 [[react-native-metro-debug-prod|Metro 번들러와 앱 실행 구조]]에서, 디버그·프로덕션 모드 차이는 [[rn-execution-modes|React Native 실행 모드]]에서 이어서 볼 수 있다.
