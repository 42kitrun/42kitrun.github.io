# 코드 난독화란? R8/ProGuard와 역난독화까지 한 번에 정리

_written by Claude & Codex_

Android 앱을 Play Console에 올리다 보면 이런 문구를 보게 된다.

> "이 App Bundle 유형과 연결된 가독화 파일이 없습니다."

처음 보면 뭔가 잘못된 것 같지만, 이 메시지는 차단 오류가 아니라 권장 알림에 가깝다. 다만 여기서 자연스럽게 질문이 이어진다. 난독화는 정확히 무엇이고, R8과 ProGuard는 어떤 역할을 하며, 왜 `mapping.txt`가 필요한가? 그리고 반대로 역공학 관점에서는 난독화된 코드를 어디까지 다시 읽을 수 있을까?

이 글은 그 흐름을 한 번에 정리한다.

---

## 먼저 구분해야 할 것: 난독화, 가독화, 암호화, 인코딩

이 주제는 용어가 섞이기 쉬워서, 시작부터 구분을 분명히 해두는 편이 좋다.

| 구분 | 난독화 | 가독화 | 암호화 | 인코딩 |
|---|---|---|---|---|
| 목적 | 코드를 읽기 어렵게 만들기 | 읽기 어려운 코드를 다시 이해하기 쉽게 풀기 | 데이터를 비밀로 보호하기 | 데이터를 다른 형식으로 표현하기 |
| 대상 | 배포 전 코드, 바이트코드, 리소스 | 난독화된 코드, 스택 트레이스, 산출물 | 평문 데이터 | 문자열, 바이너리 데이터 |
| 실행 결과 | 원래와 동일해야 함 | 실행 변경 없음, 해석 가능성만 높임 | 복호화 전에는 원래 데이터 사용 불가 | 디코딩하면 원래 데이터 사용 가능 |
| 복원 전제 | 분석과 추론이 필요 | 매핑 파일 또는 분석 도구가 필요 | 키가 필요 | 규칙만 알면 복원 가능 |
| 대표 예시 | R8, ProGuard | `mapping.txt`, retrace, 디컴파일 분석 | AES, RSA | Base64, UTF-8 |

여기서 핵심은 두 가지다.

- **난독화**는 코드를 숨기는 것이 아니라, 읽기 어렵게 만드는 변환이다
- **가독화**는 그 결과물을 다시 사람이 읽을 수 있게 풀어내는 과정이다

즉, 앱은 난독화되어 배포되고, 운영이나 분석 단계에서는 필요에 따라 가독화가 수행된다. Android에서 `mapping.txt`가 중요한 이유도 바로 여기에 있다.

---

## 난독화와 역난독화를 같이 봐야 하는 이유

![코드 난독화 개요|697](/assets/posts/security/code-obfuscation-proguard-r8/code-obfuscation-overview.svg)

위 이미지가 잘 보여주듯이, 코드 보호는 난독화만으로 끝나는 주제가 아니다. 실제 배포 환경에서는 항상 두 관점이 함께 존재한다.

- **난독화(Obfuscation)**: 배포자가 코드를 읽기 어렵게 변형하는 과정
- **역난독화(Deobfuscation)**: 분석자가 난독화된 코드를 다시 이해 가능한 형태로 최대한 복원하는 과정

즉, 난독화는 방어자의 작업이고 역난독화는 분석자의 작업이다. Android에서 R8을 이해할 때도 이 두 방향을 같이 봐야 `mapping.txt`의 의미와 한계를 정확히 이해할 수 있다.

---

## 난독화는 실제로 무엇을 바꾸는가

난독화는 "코드를 망가뜨리는 작업"이 아니라, **동작은 유지한 채 읽기 비용을 올리는 변환**이다. 대표적으로는 다음과 같은 방식이 쓰인다.

### 1. 식별자 난독화

클래스명, 메서드명, 필드명을 짧고 의미 없는 이름으로 바꾼다.

```kotlin
class PaymentProcessor {
    fun processPayment(amount: Int): Boolean {
        return amount > 0
    }
}
```

위 코드는 난독화 후 대략 이런 식으로 보일 수 있다.

```kotlin
class a {
    fun b(c: Int): Boolean {
        return c > 0
    }
}
```

코드는 여전히 실행되지만, 이름만 보고 역할을 유추하기 어려워진다.

### 2. 압축과 축소

사용하지 않는 클래스, 메서드, 리소스를 제거하고 바이트코드 구조를 더 작게 만든다. Android에서는 이 부분이 보안보다도 앱 크기 최적화에 직접적인 영향을 준다.

### 3. 코드 구조 단순화 또는 재배치

컴파일러와 최적화 도구는 인라이닝, 병합, 제거 같은 변환을 수행한다. 이 과정은 성능과 크기 개선에 유리하지만, 결과적으로 원래 소스 구조를 추적하기 어렵게 만들기도 한다.

### 4. 추가 보호 기법

개요도에 나온 것처럼 제품이나 도구에 따라 문자열 보호, 제어 흐름 교란 같은 기법이 언급되기도 한다. 다만 Android 기본 빌드 체인에서 개발자가 가장 직접적으로 마주치는 것은 **이름 축약, 미사용 코드 제거, 최적화**다.

---

## 역난독화란 무엇인가

역난독화는 난독화된 코드를 "원본 그대로 되돌리는 마법"이 아니다. 더 정확히 말하면, **분석 가능한 형태로 최대한 복원하는 과정**이다.

보통 다음 두 방향으로 이루어진다.

### 정적 분석

실행하지 않고 산출물 자체를 분석한다.

- 디컴파일 결과를 읽는다
- 호출 흐름과 패턴을 추적한다
- 클래스 관계와 조건 분기를 재구성한다
- 의미 없는 이름 대신 역할 기반 이름을 추론한다

이때 자주 쓰이는 도구로는 `jadx`, `apktool`, `Ghidra` 같은 계열이 있다.

### 동적 분석

앱을 실제로 실행하면서 동작을 추적한다.

- 런타임 메서드 호출을 본다
- 네트워크 요청과 응답을 관찰한다
- 후킹이나 디버깅으로 실제 값을 확인한다
- 메모리 상태를 확인해 정적 분석에서 놓친 맥락을 보완한다

이 흐름에서는 `Frida` 같은 도구가 자주 언급된다.

---

## 역난독화로 어디까지 복원할 수 있나

여기서 중요한 현실 인식이 하나 있다.

> 역난독화는 "가독성 회복"이지, "완전한 원본 복구"가 아니다.

복원 가능한 것과 어려운 것을 나누면 다음과 같다.

| 항목 | 복원 가능성 | 설명 |
|---|---|---|
| 호출 흐름 | 비교적 높음 | 함수 호출 관계와 실행 순서는 추론 가능 |
| 클래스 역할 | 중간 | 사용 패턴을 보면 어느 정도 유추 가능 |
| 변수명, 메서드명 | 낮음 | 원래 의미 있는 이름은 대개 사라짐 |
| 주석, 문서 | 사실상 불가 | 컴파일 산출물에 남지 않음 |
| 정확한 원본 구조 | 낮음 | 인라이닝, 병합, 최적화로 구조가 바뀜 |

그래서 난독화는 완벽한 차단책이 아니라 **분석 시간과 비용을 올리는 방어 기법**으로 이해하는 것이 맞다.

또 하나 짚어야 할 점은 법적 경계다. 정당한 보안 분석과 무단 역공학은 같은 것이 아니다. 조직이나 제품을 다룰 때는 계약, 저작권, 라이선스, 내부 정책을 먼저 확인해야 한다.

## Android에서 ProGuard와 R8은 어떤 역할을 하나

Android 실무에서는 이론보다 도구 이름을 먼저 만나게 된다. 결론부터 말하면 현재 기준으로는 **R8이 기본**이고, ProGuard는 역사적으로 함께 언급되는 이름이다.

### ProGuard

오랫동안 Android/Java 진영에서 사용된 코드 축소 및 난독화 도구다.

- 이름 축약
- 미사용 코드 제거
- 일부 최적화

### R8

Google이 Android 빌드 파이프라인에 통합한 현대적인 shrinker/optimizer/obfuscator다. 개념적으로는 ProGuard가 하던 역할을 이어받되, Android 빌드와 더 밀접하게 연결되어 있다.

실무에서는 보통 이렇게 이해하면 충분하다.

| 항목 | ProGuard | R8 |
|---|---|---|
| 위치 | 전통적인 Java/Android 난독화 도구 | Android 현재 기본 도구 |
| Android 기본값 | 과거에 주로 사용 | 현재 표준 |
| 역할 | shrink, optimize, obfuscate | shrink, optimize, obfuscate |
| 규칙 파일 | `proguard-rules.pro` 사용 | 동일하게 규칙 파일 사용 |

즉, 오늘날 Android 프로젝트에서 `proguard-rules.pro` 파일을 수정하고 있어도 실제 빌드에서 돌아가는 엔진은 R8인 경우가 대부분이다.

---

## R8이 생성하는 `mapping.txt`는 왜 중요한가

난독화를 켜면 릴리스 빌드 결과물과 함께 `mapping.txt`가 생성된다. 이 파일은 "난독화된 이름"과 "원래 이름"의 대응표다.

예를 들어:

```txt
com.example.PaymentProcessor -> a:
    boolean processPayment(int) -> b
```

이 정보가 있으면 난독화된 스택 트레이스를 다시 읽을 수 있다. 즉, Android에서의 `mapping.txt`는 보안 분석용 역난독화 도구라기보다, **운영과 디버깅을 위한 공식 복원 키**에 가깝다.

정리하면:

- 앱 배포자는 난독화된 산출물을 배포한다
- R8은 그에 대응하는 `mapping.txt`를 생성한다
- 개발팀은 이 파일로 크래시 로그를 읽을 수 있게 된다
- Play Console에 업로드하면 Android Vitals에서 사람이 읽기 쉬운 형태로 보여준다

여기서 Play Console의 "가독화 파일이 없습니다" 경고가 이해된다. 난독화를 쓰지 않았다면 `mapping.txt`도 없으니 자연스러운 상태다. 반대로 난독화를 쓰고 있다면 운영상 매우 중요한 파일이므로 관리해야 한다.

---

## Play Console 경고를 어떻게 이해하면 되나

이 문구는 흔히 오해를 부른다.

> "이 App Bundle 유형과 연결된 가독화 파일이 없습니다."

이 메시지는 보통 다음 둘 중 하나다.

| 상황 | 의미 |
|---|---|
| 난독화를 사용하지 않음 | 올릴 매핑 파일이 없으므로 자연스러운 상태 |
| 난독화를 사용함 | `mapping.txt`를 업로드하면 크래시 분석에 유리함 |

즉, 이 경고는 "보안을 위해 무조건 난독화를 켜라"는 뜻이 아니라, **난독화된 빌드를 배포한다면 그에 맞는 복원 파일도 함께 관리하라**는 운영 메시지에 가깝다.

---

## Android에서 언제 켜는 게 현실적인가

난독화는 기본적으로 릴리스 품질과 운영 편의 사이의 트레이드오프를 만든다. 그래서 프로젝트 단계별로 판단하는 게 좋다.

| 상황 | 권장 |
|---|---|
| 초기 개발, 내부 테스트 | 보통 끔. 디버깅과 원인 파악이 우선 |
| 첫 출시 직전 | 기능 안정화가 끝났다면 검토 가능 |
| 앱 크기 최적화가 중요함 | 적극 권장. shrink와 resource 최적화 효과가 큼 |
| 비즈니스 로직 보호가 중요함 | 권장. 다만 과신하면 안 됨 |
| 리플렉션, 직렬화 의존이 많음 | 규칙 점검 전에는 신중하게 적용 |

실무에서는 대개 아래 두 줄이 같이 간다.

```gradle
minifyEnabled true
shrinkResources true
```

다만 이 설정을 켜면 끝이 아니다. `-keep` 규칙이 빠지면 리플렉션, 직렬화, 어노테이션 기반 라이브러리, 네이티브 연동 지점에서 런타임 오류가 날 수 있다. 릴리스 빌드 기준 통합 테스트와 실제 시나리오 검증이 필수다.

---

## 난독화의 한계와 실무적인 결론

난독화는 분명 가치가 있다. 앱 크기를 줄이고, 코드 읽기 비용을 높이며, 단순 복제와 무차별 분석을 어렵게 만든다. 하지만 그 한계도 명확하다.

- 숙련된 분석가는 정적 분석과 동적 분석을 함께 사용한다
- 난독화만으로 핵심 비즈니스 로직을 완전히 숨길 수는 없다
- API 키나 민감 정보는 클라이언트에 두는 순간 노출 가능성을 안고 간다
- 정말 민감한 판단 로직은 서버 측으로 옮기는 것이 원칙이다

결국 좋은 전략은 하나다. **난독화는 기본 방어선으로 사용하되, 민감한 로직 보관 위치와 운영 체계까지 함께 설계해야 한다.**

---

## 정리

- 난독화는 실행 결과를 유지하면서 코드를 읽기 어렵게 만드는 변환이다
- 역난독화는 난독화된 코드를 원본 그대로가 아니라 읽을 수 있는 수준까지 복원하는 과정이다
- Android에서는 R8이 현재 기본 도구이고, `proguard-rules.pro` 규칙 체계를 계속 사용한다
- `mapping.txt`는 개발팀이 공식적으로 사용하는 역난독화 단서이며, 크래시 분석에 매우 중요하다
- Play Console 경고는 차단 오류가 아니라 운영 편의 관점의 권장 알림이다
- 난독화는 유용하지만 완벽한 보안은 아니며, 민감한 로직은 서버로 분리하는 것이 더 중요하다
