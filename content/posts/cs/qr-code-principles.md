# QR Code의 원리 — 작은 사각형이 데이터를 담는 방법

_written by Claude-Code_

회사에서 개발한 __림프부종 자가관리__ 앱과 **iCON** 의 홈페이지를 정비하면서 앱 설치 안내를 위한 QR Code를 생성했다. QR Code를 만드는 것 자체는 어렵지 않다. 하지만 그 작은 사각형 패턴 안에서 어떤 일이 벌어지는지 궁금해졌다. 왜 일부가 훼손되어도 스캔이 되는지, 어떻게 그 많은 데이터를 넣는지 — 이 글은 그 구조를 파헤친 기록이다.

---

## QR Code란?

QR Code(Quick Response Code)는 1994년 일본 덴소웨이브(Denso Wave)가 자동차 부품 추적을 위해 개발한 2차원 매트릭스 바코드다. 기존 1D 바코드가 수십 자 수준이었다면 QR Code는 수천 자까지 담을 수 있다. 수평·수직 두 축 모두 데이터를 인코딩하기 때문이다.

## 구조: 각 영역의 역할

QR Code를 현미경처럼 들여다보면 각 영역이 명확한 역할을 가진다.

```
┌─────────────────────────────────┐
│ ███  ░░░  ███  ···  ░░░  ···    │  ← Finder Pattern (3곳)
│ █ █  ░█░  █ █  ···  ░█░  ···    │
│ ███  ░░░  ███  ···  ░░░  ···    │
│                                 │
│  ·   ·   ·   ·   ·   ·   ·     │  ← Timing Pattern
│                                 │
│      [Format Info]              │  ← 오류수정레벨 + 마스크패턴
│                                 │
│      [  DATA + ECC  ]           │  ← 실제 데이터 영역
│                                 │
└─────────────────────────────────┘
```

| 영역 | 역할 |
|------|------|
| **Finder Pattern** | 세 모서리의 정사각형 패턴. 카메라가 QR Code의 위치·방향을 인식할 때 사용 |
| **Alignment Pattern** | 버전 2 이상에서 추가. 왜곡된 이미지를 보정할 기준점 |
| **Timing Pattern** | 가로·세로로 교차하는 흑백 줄무늬. 셀(모듈) 좌표 계산 기준 |
| **Format Info** | 오류 수정 레벨과 마스크 패턴 번호를 인코딩한 2개 복사본 |
| **Data + ECC** | 실제 데이터와 Reed-Solomon 오류 수정 코드 |
| **Quiet Zone** | QR Code 외곽 4셀 여백. 배경과 구분하는 역할 |

## 데이터 인코딩 흐름

QR Code 생성은 단순히 0과 1을 배치하는 것이 아니다. 다음 5단계를 거친다.

### 1단계: 인코딩 모드 선택

입력 데이터의 성격에 따라 모드를 결정한다. 더 제한적인 모드일수록 밀도가 높아진다.

| 모드 | 대상 | 효율 |
|------|------|------|
| **Numeric** | 0-9 숫자만 | 3.33 bit/char |
| **Alphanumeric** | 0-9, A-Z, 일부 기호 | 5.5 bit/char |
| **Byte (Latin-1 / UTF-8)** | 임의 바이트 | 8 bit/char |
| **Kanji** | 일본어 한자 | 13 bit/char |

URL 같은 일반 문자열은 대부분 **Byte 모드**로 인코딩된다.

### 2단계: 데이터 비트스트림 구성

```
[모드 지시자 4bit] + [문자수 필드] + [인코딩된 데이터] + [종료자 0000] + [패딩]
```

예를 들어 `https://example.com`은 Byte 모드로 인코딩되며, 각 문자를 UTF-8 바이트값으로 직렬화한 후 비트 배열로 만든다.

### 3단계: Reed-Solomon ECC 코드 생성 (→ 다음 섹션에서 상세)

데이터 비트스트림을 다항식으로 변환하고, 오류 수정 코드워드를 계산해 데이터 뒤에 붙인다.

### 4단계: 매트릭스에 배치

데이터+ECC 비트를 QR 매트릭스에 지그재그 패턴으로 채워 넣는다. 오른쪽 아래부터 시작해서 2열씩 위아래로 구불구불 올라간다.

### 5단계: 마스킹 패턴 적용 (→ 이후 섹션에서 상세)

셀이 너무 균일하면 스캐너가 오인식한다. 8가지 마스크 중 최적 패턴을 적용해 명암 균형을 맞춘다.

## ECC — 오류 수정의 핵심

QR Code가 찢기거나 더럽혀져도 스캔되는 이유는 **오류 수정 코드(ECC, Error Correction Code)** 덕분이다.

오류 수정 레벨은 4단계로 나뉜다.

| 레벨 | 기호 | 복구 가능 손상률 | 용도 |
|------|------|----------------|------|
| Low | L | ~7% | 깨끗한 환경, 데이터 밀도 우선 |
| Medium | M | ~15% | 일반 사용 (기본값) |
| Quartile | Q | ~25% | 인쇄물, 작은 QR Code |
| High | H | ~30% | 로고 삽입, 가혹한 환경 |

iCON 앱 홈페이지에 사용한 QR Code는 **Q 레벨**로 설정했다. 인쇄나 화면 출력 시 일부 픽셀이 뭉개질 수 있고, 중앙에 회사 로고를 오버레이하는 경우를 대비해 25% 복구 여유를 확보하는 것이 적합하기 때문이다.

## Reed-Solomon ECC

Reed-Solomon(RS) 코드는 1960년 어빙 리드(Irving Reed)와 구스타브 솔로몬(Gustave Solomon)이 개발한 블록 오류 수정 알고리즘이다. CD, DVD, RAID, 위성 통신, QR Code에 이르기까지 오류가 허용되지 않는 곳 어디서나 쓰인다.

### 핵심 개념: 다항식으로 데이터를 표현

RS 코드는 데이터를 **유한체(Galois Field, GF)** 위의 다항식으로 취급한다. QR Code는 GF(2⁸) — 즉 0부터 255까지의 값 256개로 이루어진 수 체계를 사용한다.

```
데이터 [D0, D1, D2, ..., Dk-1]
    ↓
메시지 다항식 M(x) = D0·x^(k-1) + D1·x^(k-2) + ... + D(k-1)
    ↓
생성 다항식 G(x) = (x - α¹)(x - α²)···(x - α^(2t))  [t: 수정 가능한 심볼 수]
    ↓
나머지 R(x) = M(x) · x^(2t)  mod  G(x)
    ↓
전송: M(x) · x^(2t) + R(x)  [이 다항식은 G(x)로 나누어 떨어짐]
```

### 왜 이것이 오류를 수정하는가?

수신 측은 받은 다항식을 G(x)로 나눈다. 오류가 없으면 나머지가 0이다. 나머지가 0이 아니면 **신드롬(syndrome)**을 계산해 오류 위치와 크기를 역산한다.

M 레벨 기준으로 Version 1 QR Code는 데이터 코드워드 19개, ECC 코드워드 7개를 가진다. 즉 7개 코드워드가 손상되어도 원본을 복원할 수 있다.

## 마스킹 패턴

데이터를 배치한 QR 매트릭스가 특정 패턴 — 예를 들어 세로 줄무늬만 가득하거나 전체가 흰색이면 — 스캐너가 Finder Pattern이나 Timing Pattern을 오인식할 수 있다. **마스킹**은 이런 문제를 방지하는 후처리 단계다.

8가지 마스크 패턴 중 하나를 XOR 연산으로 데이터 영역에 적용한다.

| 패턴 번호 | 조건 (해당하면 반전) |
|-----------|---------------------|
| 0 | `(i + j) % 2 == 0` |
| 1 | `i % 2 == 0` |
| 2 | `j % 3 == 0` |
| 3 | `(i + j) % 3 == 0` |
| 4 | `(i/2 + j/3) % 2 == 0` |
| 5 | `(i*j) % 2 + (i*j) % 3 == 0` |
| 6 | `((i*j) % 2 + (i*j) % 3) % 2 == 0` |
| 7 | `((i+j) % 2 + (i*j) % 3) % 2 == 0` |

QR Code 생성기는 8가지 마스크를 모두 적용해보고 **패널티 점수**가 가장 낮은 것을 선택한다. 패널티는 연속된 동일색 모듈, 비율 불균형, Finder Pattern 유사 패턴 등을 기준으로 계산된다.

## 버전과 용량

QR Code는 **버전 1부터 40**까지 존재한다. 버전 N의 크기는 `(4N + 17) × (4N + 17)` 셀이다.

| 버전 | 크기 | Byte 모드 최대 용량 (L 레벨) |
|------|------|---------------------------|
| 1 | 21 × 21 | 17자 |
| 5 | 37 × 37 | 106자 |
| 10 | 57 × 57 | 271자 |
| 20 | 97 × 97 | 858자 |
| 40 | 177 × 177 | 2,953자 |

URL 하나 담는 데는 Version 3~5면 충분하다. iCON 앱의 App Store / Play Store 링크처럼 약 80~100자짜리 URL은 Q 레벨 기준 Version 5~6이면 안정적으로 인코딩된다.

용량을 늘리려면 버전을 올려야 하지만, 그만큼 셀 수가 늘어나 QR Code 크기도 커진다. **데이터는 짧게, ECC 레벨은 필요한 만큼** — 이 균형이 실무 핵심이다.

## 실무 관점 요약

QR Code를 앱 다운로드 유도에 활용할 때 실제로 결정해야 할 것은 두 가지다.

**1. 무엇을 URL로 쓸 것인가?**

App Store / Play Store 직접 링크보다 **자체 리다이렉트 URL**이 낫다. 이유는 세 가지다.

- URL이 바뀌어도 QR Code를 다시 인쇄하지 않아도 된다
- 리다이렉트 전 User-Agent 기반으로 iOS/Android 분기 처리가 가능하다
- 접속 통계를 수집할 수 있다

**2. ECC 레벨을 얼마로 설정할 것인가?**

| 상황 | 권장 레벨 |
|------|----------|
| 웹/앱에서 화면 표시만 | M |
| 인쇄물, 포스터 | Q |
| 로고 삽입 오버레이 | H |
| 야외 환경, 손상 가능성 | H |

iCON의 경우 홈페이지 화면 표시 + 인쇄 배포 가능성 모두를 고려해 Q 레벨로 결정했다.

---

## 정리

QR Code는 단순한 픽셀 패턴이 아니다. 데이터 인코딩 → RS 오류 수정 코드 생성 → 매트릭스 배치 → 마스킹이라는 정밀한 파이프라인의 결과물이다. 30%까지 손상돼도 복원되는 이유는 RS 코드의 수학적 견고함 덕분이고, 어떤 각도에서도 인식되는 이유는 Finder Pattern과 Timing Pattern의 설계 덕분이다.

앱 설치 유도 QR Code 하나를 만들면서 이 구조를 뜯어보고 나니, 작은 네모 코드 뒤에 얼마나 많은 공학적 판단이 숨어 있는지 새삼 실감하게 됐다.
