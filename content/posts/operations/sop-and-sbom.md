# SOP와 SBOM — 표준 운영 절차와 소프트웨어 자재 명세서

_written by Claude-Code_

둘 다 "무엇을, 어떻게 관리할 것인가"에 대한 답이지만, 대상이 다르다.

```
SOP  → 사람이 따르는 절차 문서   "어떻게 한다"
SBOM → 소프트웨어 구성 목록      "무엇이 들어있다"
```

---

## 1. SOP (Standard Operating Procedure) — 표준 운영 절차

### 개념

특정 작업을 **일관되고 안전하게 수행하기 위한 단계별 절차 문서**.
담당자가 바뀌어도 같은 품질과 결과가 나오도록 표준화한다.

```
문제 발생
    │
[SOP 없을 때]              [SOP 있을 때]
    │                           │
담당자마다 다른 방식        정해진 절차를 따름
오류 발생 가능성 높음       일관된 결과 보장
재현·학습 어려움            누구든 동일하게 수행 가능
```

### SOP의 구조

```
SOP 문서
├── 제목 / 버전 / 작성일
├── 목적 (Purpose)         ← 왜 이 절차가 필요한가
├── 적용 범위 (Scope)      ← 누가, 어떤 상황에서 사용하는가
├── 역할과 책임 (Roles)    ← 담당자 명확화
├── 선행 조건              ← 시작 전 준비 사항
├── 단계별 절차            ← Step 1, Step 2, Step 3...
├── 예외 처리              ← 비정상 상황 대응
└── 관련 문서 / 참고
```

### 소프트웨어 분야의 SOP 예시

| 분야 | SOP 예시 |
|------|----------|
| 배포 | 프로덕션 배포 절차 (승인 → 빌드 → 배포 → 검증 → 롤백 기준) |
| 보안 | 취약점 발견 시 대응 절차 (탐지 → 격리 → 분석 → 패치 → 보고) |
| 장애 | 인시던트 대응 절차 (감지 → 에스컬레이션 → 조치 → 사후 검토) |
| 온보딩 | 신규 개발자 환경 세팅 절차 |

### SOP가 필요한 이유

```
항공·의료·군사 → 오래전부터 SOP 필수
       │
       ↓
소프트웨어 산업도 복잡성 증가로 동일한 필요성
       │
       ├── 규정·컴플라이언스 요구 (ISO, SOC2, GDPR)
       ├── 장애 시 일관된 대응
       └── 지식의 개인 의존 탈피
```

## 2. SBOM (Software Bill of Materials) — 소프트웨어 자재 명세서

### 개념

소프트웨어에 포함된 **모든 구성 요소(라이브러리·패키지·의존성)의 목록**.
제조업의 BOM(부품 명세서)을 소프트웨어에 적용한 개념.

```
my-app v1.0
├── react 18.2.0          (MIT)
│   └── scheduler 0.23.0  (MIT)
├── express 4.18.2        (MIT)
│   └── qs 6.11.0         (BSD-3) ← 취약점 CVE-2022-24999
└── lodash 4.17.21        (MIT)
```

SBOM이 있으면 취약한 컴포넌트를 **즉시 조회·추적** 가능.

### 왜 필요한가 — Log4Shell 사례 (2021)

```
log4j 취약점 발견
        │
        ├── 문제: log4j가 어디에 쓰였는지 아무도 몰랐음
        ├── 결과: 수천 개 서비스가 자신이 취약한지 확인조차 불가
        └── 소요: 패치까지 몇 주 ~ 몇 달

SBOM이 있었다면 → "log4j 2.x 사용 서비스" 즉시 목록화 → 빠른 대응
```

### SBOM 포함 정보

| 항목 | 예시 |
|------|------|
| 패키지명 | lodash |
| 버전 | 4.17.21 |
| 라이선스 | MIT |
| 출처 | npm registry |
| 해시값 | sha256:abc123... |
| 직접/간접 의존 | 직접 |

### 주요 포맷

```
SPDX      → Linux Foundation 주도, ISO 국제 표준 (ISO/IEC 5962)
CycloneDX → OWASP 주도, 보안·취약점 연계에 강점
```

### SBOM 생성·활용 흐름

```
소스코드 / 빌드 결과물
        │
   [SBOM 생성 도구]
   syft / cdxgen / trivy
        │
   SBOM 파일 (JSON/XML)
        │
   ┌────┴──────────────┐
   │                   │
취약점 스캔          라이선스 검토
(CVE DB 연동)       (GPL 충돌 여부 등)
```

## SOP와 SBOM의 관계

```
취약점 발견 (예: 사용 중인 라이브러리에서 CVE 발견)
        │
   SBOM 조회          ← "어떤 서비스가 영향받는가?" 파악
        │
   SOP 실행           ← "어떤 순서로 대응하는가?" 절차 수행
        │
   패치 → 검증 → 배포 → 보고
```

SBOM은 **현황 파악** 도구, SOP는 **행동 지침** 도구.
둘이 함께 있을 때 보안 대응 속도와 품질이 올라간다.

## 한눈에 비교

| 구분 | SOP | SBOM |
|------|-----|------|
| 정의 | 표준 운영 절차 문서 | 소프트웨어 구성 요소 목록 |
| 형태 | 텍스트 문서 (절차서) | 구조화 데이터 (JSON/XML) |
| 목적 | 일관된 작업 수행 | 구성 요소 투명성 확보 |
| 생성 주체 | 사람 (팀·조직) | 자동화 도구 |
| 관련 규제 | ISO, SOC2, GDPR | 미국 EO 14028, EU CRA |

---

## 실무 포인트

**SOP**
- 작성보다 **최신화**가 어렵다 — 버전 관리와 정기 리뷰 일정을 반드시 포함
- 실제 사용하지 않는 SOP는 없는 것과 같다 — 짧고 실용적으로 유지

**SBOM**
- CI/CD 파이프라인에 SBOM 생성 단계를 추가한다 (빌드할 때마다 갱신)
- SBOM을 CVE DB([[dos-ddos-attacks|보안 공격 분류]])와 연동해 자동 스캔
- 오픈소스 사용 시 GPL ↔ 상업 라이선스 충돌 여부 반드시 확인
