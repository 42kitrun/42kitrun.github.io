# 서명과 암호화는 다르다

두 개념은 자주 혼용되지만 목적이 완전히 다르다.

| | 암호화 | 서명 |
|---|---|---|
| **목적** | 내용 숨기기 | 출처·무결성 증명 |
| **결과** | 원본 읽을 수 없음 | 원본 그대로, 서명값만 추가 |
| **키** | 수신자 키로 잠금 | 발신자 키로 서명 |

![서명 vs 암호화 개념 비교](/assets/posts/security/signing-vs-encryption/signing_vs_encryption.svg)

---

## 암호화 — 내용을 잠근다

원본 데이터를 키로 뒤섞어 **읽을 수 없는 암호문**으로 만든다.

```
"Hello World"  →  AES/RSA  →  Xk92mP#!...  (판독 불가)
```

복호화 키 없이는 내용을 볼 수 없다. 내용 기밀이 목적이다.

---

## 서명 — 지문을 붙인다

원본은 그대로 두고, **해시값에 비밀키를 적용한 서명**을 덧붙인다.

```
"Hello World"  +  hash_hmac(data, secret)  →  "Hello World" + {서명값}
```

내용은 누구나 볼 수 있다. 대신 이 서명을 검증하면 **누가 만들었는지, 중간에 변조됐는지** 확인할 수 있다.

> 서명은 내용을 숨기지 않는다. "내가 만든 것, 아무도 안 건드린 것"만 증명한다.

---

## 실제 사례 3가지

![JWT, AAB, SSL 세 가지 서명 사례 비교](/assets/posts/security/signing-vs-encryption/three_signing_cases.svg)

### JWT — 토큰 위변조 방지

```
header.payload.signature
```

payload는 base64로 인코딩되어 있어 **누구나 디코딩해서 볼 수 있다.** 서명은 이 토큰을 서버가 직접 발급했는지, 클라이언트가 payload를 몰래 바꾸지 않았는지 검증하는 데 쓰인다. 내용 보호가 아니라 위변조 탐지가 목적이다.

- 방식: HMAC (대칭키) — 발급·검증 모두 서버가 같은 secret으로 처리
- 민감 정보를 payload에 넣으면 안 되는 이유가 여기에 있다

### AAB / APK 서명 — 앱 출처 증명

앱 코드는 공개된다. 서명은 "이 앱이 정말 그 개발자가 빌드한 것인지"를 Android OS가 설치 시점에 검증하는 데 쓰인다. 서명이 다르면 같은 패키지명이라도 업데이트가 거부된다.

- 방식: RSA (비대칭키) — 개발자 비밀키로 서명, OS가 공개키로 검증

### SSL 인증서 서명 — 사이트 신원 확인

인증서에는 도메인 이름과 공개키가 담겨 있고, CA(인증기관)가 그 내용을 서명한다. 브라우저는 CA의 공개키로 서명을 검증해 "이 인증서가 신뢰할 수 있는 기관이 발급한 것"임을 확인한다. HTTPS 연결 시 실제 데이터 암호화는 이 검증 이후에 별도로 이루어진다.

- 방식: RSA (비대칭키) — CA 비밀키로 서명, 브라우저가 공개키로 검증

## 정리

- **암호화가 필요할 때**: 내용 자체를 숨겨야 할 때 (메시지 전송, 파일 보호)
- **서명이 필요할 때**: 출처를 증명하거나 변조를 탐지해야 할 때 (토큰, 앱 배포, 인증서)
- **둘 다 필요할 때**: HTTPS — 인증서 서명으로 신원 확인 후, 대칭키 암호화로 데이터 보호
